android ipc 学习笔记 2 使用 aidl 进行进程间通信进阶

2016-06-14

这篇文章是Android IPC 的第二篇学习笔记，主要内容是

  • 客户端与服务端的工作线程
  • 多进程之间的观察者模式
  • 服务意外中止的监听和重新连接
  • 使用封装过后的 Messenger 进行进程间通信

客户端与服务端的工作线程

  • 在客户端与服务端绑定成功之后，会调用 onServiceConnected() 方法，这个方法是运行在 UI 线程中的，如果服务端有一个耗时的方法，那么我们不能直接调用这个耗时方法。

  • 服务端的方法本身不在 UI 线程中执行，所以在服务端，则不必开一个子线程去执行耗时的任务。

多进程之间的观察者模式

观察者模式在 Android 开发中很常见，平时我们用到的一大堆 Listener 或者是 CallBack 都是观察者模式的应用，在多进程的通信中，我们有时候也需要完成以下需求场景

服务端中运行某一个功能，客户端需要订阅其结果

比如

服务端下载一个文件，客户端需要订阅下载的结果，来处理 UI

那么，在单个进程的情况下，我们很容易做到，只需要一个方法即可

但是，考虑到不同进程之间，并不共享内存区域，也就是一个进程无法直接传递一个对象到另外一个进程，那么两个进程各自持有的观察者并非同一个实例，假如客户端需要取消订阅任务的结果，那么服务端进行移除操作也只能移除调服务端的观察者实例，而非客户端的观察者实例。

Android 官方提供了一个类以帮助开发者完成观察者模式的实现。

RemoteCallbackList<E extends IInterface>

这个类内部维护了一个 HashMap<< IBinder, Callback > 也就是说它将 IBinder 作为 key，来确保能够准确定位到不同进程间的观察者实例（服务端和客户端的 IBinder 是同一个实例）

由于 AIDL 接口传递的数据类型限制，泛型中的 E 也是需要是 AIDL 接口。

具体使用：

  • 注册一个观察者

RemoteCallbackList<E extends IInterface>.register(T e);

  • 移除一个观察者

RemoteCallbackList<E extends IInterface>.unregister(T e);

  • 遍历访问观察者

服务意外中止的监听和重新连接

有两种方式可以监听到服务的意外中止

  • 一种是设置 DeathRecipient，运行在 Binder 线程池中，不能直接访问 UI
  • 一种是在 onServiceDisconnected 中进行处理，运行在 UI 线程中

使用封装过后的 Messenger 进行进程间通信

Android 封装了一个 Messenger 类，可以简化 AIDL 的通信，以下是一些要点。

  • 一般 Messenger 需要一个 Handler 的实例或者 Handler 子类的实例，用以处理消息。

  • 服务端在 IBinder onBind(Intent intent) 方法中返回 Messenger 内部的 Binder 对象

  • 客户端在 onServiceConnected(ComponentName name, IBinder service) 中实例化 Messenger

  • 使用 Messenger 可以发送消息，本质上是内部的 Handler 在发送消息

  • Message 内部有一个成员变量为 Messenger 类型，这个成员变量可以回复消息，回复的消息会被它内部的 Handler 实例处理。

  • 所以客户端如果要得到服务端的回应，可以声明一个 Handler，并且构造 Messenger 作为 Message 的 replyTo 变量。

  • 服务端获取到 replyTo 变量之后，用它来向客户端发送消息。
