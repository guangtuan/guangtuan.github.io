编译时的工作，生成 ProxyInterface 和 Proxy

既然在工程的 gradle 文件中声明了插件和编译路径

那么很明显，Realm 是在编译时候做了工作的，我们可以在 app/build/generated/source/apt/debug 中找到痕迹。

毫无疑问，PersonRealmProxyInterface.java 和 PersonRealmProxy.java 是 Realm 在编译时生成的文件。

对 ProxyInterface 和 Proxy 进行分析

ProxyInterface

PersonRealmProxyInterface 是接口的声明，包含了 get 方法和 set 方法。这里的 get 方法和 set 方法代理了对应 model 的 get 方法和 set 方法，稍后会给出说明。

Proxy

PersonRealmProxy 是接口的实现，同时实现了很多通用的方法，Realm 对象会在操作 Model 的时候，调用到这些方法。

  • createOrUpdateUsingJsonObject()
  • createUsingJsonStream()
  • copyOrUpdate()
  • copy()
  • createDetachedCopy()
  • …more

PersonRealmProxy.java 的声明：

这个类继承了 Person，并且实现了动态生成的 PersonRealmProxyInterface.java 接口和这个接口代理的 set 方法和 get 方法。

而 另外一个实现的接口，RealmObjectProxy.java 是一个空接口，用来更方便地标示这个类是 Realm 在编译时候产生的，当然，在 Java 8 的环境下，可以将通用的静态方法放在这个空接口中。

空接口在这里的应用也给了我们一点启示，

如在 Android 中里，如果是遵循标准的 ActionBar 开发的二级页面，一般会把 ActionBar 的左上角设置为返回的箭头，这里给出一个思路。

这样子，只要是二级页面，我们只需要实现 SecondPage.java 这个空接口就行了。

Model 和 ModelRealmProxy 之间的调用

以 createObject() 方法为例，分析 Realm 操作 Model 的时候，具体的调用过程。

第一行是检查是否合法，第二行是将对 Person 操作转换成 PersonRealmProxy 的重要操作。

其中configuration.getSchemaMediator().getTableName(clazz) 是关键部分，这里的 SchemaMediator 是一个重要角色，如果不知道，就得先看看基础使用啦。

SchemaMediator 是一个 abstract 类，Realm 在编译的时候给出了默认实现。

DefaultRealmModuleMediator.java

这里的 getTable() 方法，直接就将 Person 和 PersonRealmProxy 对应起来了。

再回来看 createObject() 方法，这里返回的是 get(clazz, rowIndex)

来看这个 get(clazz, rowIndex) 方法

同样是 SchemaMediator 类的默认实现 DefaultRealmModuleMediator.java，我们看 newInstance() 方法:

所以，Realm 以 Person.class 为参数创建实例的时候，返回的时候都是对应的 Proxy 类的实例。

所以，Realm 在整个操作 Model 的过程中，都使用了 Model 的 Proxy 类，这样子最大程度的减少了对 Model 代码的入侵，只需要继承 RealmObject，即可完成所有的操作，一切都在编译时产生，也不会影响运行时的效率。
