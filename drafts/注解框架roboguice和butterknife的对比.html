注解框架RoBoGuice和ButterKnife的对比

注解框架RoBoGuice和ButterKnife的对比

前言

RoBoGuice 和 ButterKnife 都是 Android 开发中常用的运行时注解框架，可以提高开发速度，帮助开发者省去重复繁琐的步骤，让代码更加优雅，这篇文章的主要内容是两者之间的区别。

RoBoGuice 的功能概览

注：以下功能都需要在 RoboActivity 的子类或者 RoboFragment 的子类中使用。

  • 布局
    • @ContentView(layoutID)
  • 视图
    • @InjectView(viewID) View view
  • 资源文件
    • @InjectView(resourceID) Resource res
  • 系统服务
    • @Inject SystemService systemService
  • POJO 的构造函数
    • @Inject
  • POJO
    • @Inject Foo foo;
    • 需要先注解对应的　POJO　类的构造函数
  • Application 单例
    • @Singleton
    • 随应用结束而销毁
  • Context 单例
    • @ContextSingleton
    • 当 Context 对象被回收，单例一起被回收
  • Fragment 单例
    • @FragmentSingleton
    • 当 Fragment 对象被回收，单例一起被回收
  • 接口，抽象类的实现类
    • @Inject IFoo foo
    • 需要在 AndroidManifest.xml 文件中注册自定义的模块，该模块需要继承自 AbstractModule，调用 bind(IFoo.class).to(Foo.class) 方法绑定接口（或者抽象类）和实现类，题外话，这点和 Laravel 的依赖注入方式有点类似啊
  • Service
    • 继承 RoboService, 在 Service 中使用 RoBoGuice 的其他注解
  • BroadcastReceiver
    • 不需要改变继承结构,在 BroadcastReceiver 中自由地使用 RoBoGuice 的其他注解，但是需要注意，在 BroadcastReceiver 中的 Context 或者 Fragment 单例都会被转换成全局单例
  • 单元测试
    • 官方文档
  • 自定义 View
    • 在构造函数中调用 RoboGuice.getInjector(getContext()).injectMembers(this); 以告知 RoboGuice 处理该类的注解
    • 实现方式与 POJO 中的注解没有区别
  • ContentProvider
    • 官方文档
    • 使用 AbstractModule 子类，@Provides 注解, @Named 提供特定的 Uri 实例
    • 需要在 AndroidManifest.xml 文件中声明 provider
  • 注入事件
    • 官方文档
    • 在同一个 Context 或者不同的 Context 中传递和处理事件
  • 日志
    • Ln.v(), Ln.d() 等
    • 会打印出一些日志之外的信息，如时间戳，线程信息和其他有用的信息，是一个非常棒的 Log 辅助

ButterKnife 的功能概览

注：需要调用 ButterKnife.bind(target) 进行绑定;

  • 视图
    • @BindView(viewID) View view
  • 多个视图
    • @BindViews({ viewID, viewID...}) List<View> views;
  • 字符串资源
    • @BindString(stringID) String string
  • 图片资源
    • @BindDrawable(drawableID) Drawable drawable
    • 其他资源如 Dimen, Color类似
  • 对象属性
    • ButterKnife.apply(view, ButterKnife.Action, value)`
    • 其中ButterKnife.Action<View> 也可以是 Android　原生 API 的中的 Property
  • 点击事件
    • @OnClick(viewId) void onClick(View view)
    • 同样可以绑定多个视图到一个方法中
    • 同时 OnItemSelected OnCheckedChanged…　等其他的事件都可以绑定，具体见 官方文档

RoBoGuice 小结

  • RoBoGuice 的功能非常全面，可以绑定视图，资源文件，Android 四大组件，Fragment，日志，单元测试等，此外还包含有单例对象，以声明的方式指定接口的实现类，甚至能够绑定组件的生命周期事件，以及开发者自定义的事件。

  • 在开发过程中，会变得很少看到构造函数，或者是 getSystemService() 之类的方法，也可以省去很多空值检查，因为 RoBoGuice 将这些重复的步骤都抽象起来，放在了 RoboActivity 或者 RoboFragment 中实现，开发者只要继承特定的父类以及加上特定的注解，就可以省去这些繁琐的业务无关的代码。

  • 因此，RoBoGuice 对代码的入侵也非常大，基本上囊括了 Android 开发中需要的所有的辅助功能。

    • 绑定视图代替 findViewById()，
    • 使用 AbstractModule 绑定接口的实现类可以代替工厂等，
    • 绑定了各种事件，也可以将项目中的 EventBus 去掉了,

  • 占据父类，Activity 和 Fragment 都需要继承特定的父类。

ButterKnife 小结

  • ButterKnife 的功能侧重点在于视图，和视图的属性，事件，对于其他则并不关心，对代码的入侵非常小，因此可以和其他的框架组合在一起使用。

总结

可以看出，RoBoGuice 相比 ButterKnife，功能多出不少，甚至可以说后者只是前者的一小部分，但是 ButterKnife 在视图这一块做的比 RoBoGuice 更加的细致。

两个框架的哲学也不尽相同，RoBoGuice 控制着整个应用的整体架构，充满了面向对象，依赖注入的思想。而 ButterKnife 更像是一个精细的辅助小工具。

附：题外话，RoBoGuice 的依赖注入和 Laravel 的依赖注入的比较

RoBoGuice 的依赖注入：

  • 接口和实现类的声明：
  • 在 AbstractModule 中绑定接口和实现类
  • 在 AndroidManifest.xml 中声明这个模块

Laravel 的依赖注入：

  • 接口和实现类的声明
  • 在 ServiceProvider 中进行绑定
  • 在 config/app.php 中声明该 Provider
